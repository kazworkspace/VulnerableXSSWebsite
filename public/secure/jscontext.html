The Fix — safeJsonForScript()

🟢 Two layers of protection
function safeJsonForScript(value) { return JSON.stringify(value) // Layer 1: JSON.stringify() wraps in quotes + escapes " as \" // Input: lightblue"; alert(1);// // Output: "lightblue\"; alert(1);//" ← " escaped, string intact // Layer 2: encode < > / to Unicode escapes // Prevents </script> from closing the <script> block early .replace(/</g, '\u003c') // < → < .replace(/>/g, '\u003e') // > → > .replace(/\//g, '\u002f'); // / → / // HTML parser never sees </script> — block stays open ✅ // JS engine decodes < back to < — value stays correct ✅ }
ⓘ
JSON.stringify() alone is not enough — it does not encode /, so </script> in a value would still close the script tag. The extra .replace() calls prevent this.

Why escapeHtml() Is the Wrong Fix Here

🔴 Context mismatch — wrong tool for the job
// Suppose someone "fixes" it with escapeHtml() instead: const color = escapeHtml(req.query.color); // Input: lightblue"; alert(1);// // Output: lightblue&quot;; alert(1);// // Rendered inside <script>: var bgColor = "lightblue&quot;; alert(1);//"; // In a <script> block, HTML entities are NOT decoded by the JS engine. // &quot; = literal 6-character string, NOT a quote → string does close ✅ // ...but escapeHtml() is still the wrong tool because: // ❌ Single-quote context: var x = '${color}'; // payload: '; alert(1);// → escapeHtml encodes " not ' → breaks out // ❌ No-quote context: var x = ${color}; // any JS expression works — not a string at all // ❌ Template literal: var x = `${color}`; // payload: ${alert(1)} → escapeHtml does not encode $ { } // ✅ JSON.stringify() is designed for this — works for all cases above.

Cleanest Alternative — data-* Attribute

📜 Pass server data to JS via DOM, not inline script
<!-- Server renders — escapeHtml() for HTML attribute context --> <div id="app" data-color="${escapeHtml(color)}"></div> <!-- Client JS reads it safely --> <script> // ✅ Reads from DOM attribute — no script injection possible const color = document .getElementById('app') .dataset.color; document.body.style.background = color; </script>
ⓘ
Cleanest approach — data travels as an HTML attribute (HTML-escaped at render time), never embedded in a script block. Zero risk of JS string injection from server-side rendering.
