Identifying XSS Injection Points

Before you can test for XSS, you need to find every place where user-controlled input flows into the page output. This is reconnaissance. Miss an input — miss a vulnerability.

What Makes an Input a Candidate?

💡 The Core Rule

Any input that is later reflected back in an HTML response — whether immediately, after storage, or via JavaScript — is a potential XSS injection point. The question is how the value lands in the page.

🔎 What to Look For

Submit a unique marker like xsstest1234 in every field. Then view the page source and search for your marker. If it appears — unencoded or partially encoded — you found an injection point.

🎯 Injection Context Matters

The location where your input appears determines the payload. Inside an HTML tag body? Inside an attribute? Inside a <script> block? Inside a URL href? Each needs a different attack.

⚠ Hidden Inputs Too

Developers often forget type="hidden" fields, HTTP headers (User-Agent, Referer), URL fragments (#hash), and JSON API responses rendered via innerHTML. Test all of them.

Injection Points Outside the Form

🔗 URL Parameters — Always in Scope

Every query string parameter is a potential injection point, even if no visible form controls it. Developers often add parameters for debugging, pagination, or feature flags and forget they are reflected.

http://localhost:3001/profile?search=alice&page=1&sort=name&debug=true

Test each: replace the value with xsstest1234, submit, view source, search for your marker. Try each known-dangerous parameter name: q, query, search, redirect, url, next, return, ref, lang, id, name, msg, error.

🔗 URL Fragment (#hash) — DOM XSS Only

The fragment (#...) is never sent to the server — it only exists in the browser. If JavaScript reads location.hash or location.href and writes it to innerHTML, document.write(), or eval(), you have DOM XSS. Server logs won't show this. WAFs won't see it.

http://localhost:3001/page#<img src=x onerror=alert(1)>
^ fragment never reaches server — purely client-side

📧 HTTP Headers — Require curl or Burp

Some applications reflect HTTP headers into the response (logging, analytics, diagnostics). These can't be controlled from an HTML form — use curl or Burp Suite Repeater.

Headers to test:

  • User-Agent — often logged and displayed in admin panels
  • Referer — shown in analytics dashboards
  • X-Forwarded-For — reflected in IP logging pages
  • Accept-Language — locale selection sometimes reflected
  • Cookie values — reflected in personalization messages

curl command:

curl http://localhost:3001/api/challenges/6 \
  -H 'User-Agent: <script>alert(1)</script>'

Reconnaissance Checklist

  • R Submit xsstest1234 in every visible text field and search for it in the page source. Any unencoded match is a reflected XSS candidate.
  • R Check every URL query parameter — even ones not controlled by a visible form. Try swapping the value to your marker.
  • S Identify fields that are saved and displayed to other users (usernames, bios, comments, forum posts). These are Stored XSS candidates with higher impact.
  • U Find any field whose value goes into an href, src, action, or formaction attribute. Test javascript:alert(1) — HTML escaping won't block it.
  • D Check the URL fragment (#...). Open DevTools Console and inspect whether any JS reads location.hash and writes it to the DOM.
  • H View page source and look for <input type="hidden"> fields. Edit their values in DevTools before submitting.
  • H Use curl with custom headers (User-Agent, Referer, X-Forwarded-For) and your marker value. Check if any appear in the HTML response.
  • D Search JavaScript source files for dangerous sinks: innerHTML, document.write(), eval(), location.assign(), setTimeout(string). Trace what data flows into them.
