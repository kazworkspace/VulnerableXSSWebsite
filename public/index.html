💾 Stored XSS

Payload is saved to the database and served raw to every visitor. Most dangerous — no interaction needed beyond normal browsing.

📄 DOM-based XSS

Payload stays in the URL fragment (#...) — never sent to the server. Client-side JS reads it and writes to innerHTML. Invisible to server logs and WAFs.

Vulnerable Secure

🔌 Attribute Context XSS

Input lands inside HTML attributes: unquoted attribute injection, quoted breakout via ", and javascript: URL in href. Three sub-cases, three different fixes.

💻 JS Context XSS

Input lands inside a <script> block. escapeHtml() is the wrong fix here. Correct tool: JSON.stringify() + Unicode encoding for </script> injection.

How Each Type Works

Attack flow comparison

  • R
    Reflected: Crafted URL → victim clicks → server embeds raw input in response → browser executes as trusted-site script.
  • S
    Stored: Attacker submits payload → stored raw in DB → every page load serves script to all visitors → browser executes.
  • D
    DOM-based: Payload in #fragment → client JS reads it → innerHTML executes it. Server never sees the payload — bypasses WAFs and server logs.
  • A
    Attribute context: Payload injected into HTML attribute — breaks out of the attribute via " or injects javascript: URL. escapeHtml() fixes A+B but NOT C (href scheme injection).
  • J
    JS context: Payload inside a <script> block — " closes the string literal, rest executes as code. Fix is JSON.stringify() + encode <>/, not escapeHtml().

Defense Summary

Injection Context Correct Defense
HTML body content escapeHtml(value)
HTML attribute (always quote!) escapeHtml(value)
URL attribute (href, src, action) escapeHtml(safeUrl(value))
Inside <script> block safeJsonForScript(value)
URL query parameter encodeURIComponent(value)
Rich text (user HTML) DOMPurify.sanitize(value)
