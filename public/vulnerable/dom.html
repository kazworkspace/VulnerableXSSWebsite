DOM-based XSS

Payload lives in the URL fragment (#...). Client-side JavaScript reads it and writes to innerHTML — the server never sees the payload, making it invisible to server-side defenses.

⚠
Root cause: location.hash is read by client JS and injected via innerHTML. The browser parses and executes any HTML/JS in it. The hash is never sent to the server — no logs, no WAF.

Code — Vulnerable vs Secure

🔴 VULNERABLE — this page
const name = decodeURIComponent( location.hash.slice(1) ); // ❌ innerHTML parses raw HTML document .getElementById('greeting') .innerHTML = `Hello, ${name}!`;
⚠ <img onerror=...> in the hash becomes a real element — event fires.
🟢 SECURE — /secure/dom.html
const name = decodeURIComponent( location.hash.slice(1) ); // ✅ append() creates Text nodes — no HTML parsing const el = document .getElementById('greeting'); el.replaceChildren(); el.append('Hello, ', name, '!');
✓ Strings in append() are Text nodes — < never parsed as HTML.

Attack Flow — Why DOM XSS Is Stealthy

  • 1
    Attacker crafts URL: /vulnerable/dom.html#<img src=x onerror=alert(document.cookie)>
  • 2
    URL fragment (#...) is never sent to the server — no server logs, no WAF sees it.
  • 3
    Victim's browser loads the page. Server returns the normal, clean HTML.
  • 4
    Client JS reads location.hash and writes it to innerHTML.
  • 5
    Browser executes the injected element's event handler. Cookie theft complete. Attack is invisible server-side.

DOM XSS — Dangerous Sinks

🔴 Avoid these with user-controlled input
// ❌ HTML-parsing sinks el.innerHTML = userInput; el.outerHTML = userInput; document.write(userInput); el.insertAdjacentHTML('...', userInput); // ❌ JS-execution sinks eval(userInput); setTimeout(userInput, 0); // string form only new Function(userInput)(); // ❌ Navigation / URL sinks location.href = userInput; // javascript: URL el.src = userInput;
