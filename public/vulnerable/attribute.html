Attribute Context XSS

User input lands inside HTML attributes — three distinct injection patterns. escapeHtml() alone does not fix all of them. Each requires a specific defense.

⚠
Key insight: The right defense depends on the injection context. Same function, wrong context = still vulnerable.

Case A — Unquoted Attribute

🔴 Vulnerable

Value injected without quotes: <div style="background:COLOR">
A space in the payload ends the attribute and starts a new one — injects event handler.

💡 Hover the card after submitting — event handler fires on mouse enter.

Case B — Quoted Attribute Breakout

🔴 Vulnerable

Value inside quotes but not escaped: <input value="USERNAME">
A " in the payload closes the attribute — the rest becomes new HTML attributes.

Click to load payload into Case B field

💡 Input auto-focuses on page load when the payload is active.

Case C — href javascript: Injection

🔴 Vulnerable

User-supplied URL placed directly in href.
javascript:alert(1) has no HTML special characters — escapeHtml() passes it unchanged. Browser executes it as JavaScript on click.

Click to load payload into Case C field

💡 Click "Visit website" on the result page — javascript: URL executes.

Case D — Unquoted data-* Attribute

🔴 Vulnerable

Toolbar selector injected into an unquoted data-toolbar attribute:
<table data-toolbar=SELECTOR>
No quotes around the value — a space ends the attribute and starts a new one. Payload: #t onmouseover=alert(1) injects a mouseover handler.

Code — Why Each Case Is Vulnerable

🔴 VULNERABLE — /api/profile route
// Case A — no quotes, no escaping `<div style="background:${color}">` // payload: red" onmouseover="alert(1) // result: <div style="background:red" onmouseover="alert(1)"> // Case B — quoted but not escaped `<input value="${username}">` // payload: " autofocus onfocus="alert(1) // result: <input value="" autofocus onfocus="alert(1)"> // Case C — no URL scheme check `<a href="${website}">Visit</a>` // escapeHtml("javascript:alert(1)") = "javascript:alert(1)" — unchanged! // No HTML special chars in javascript: URLs. // Case D — unquoted data-* attribute `<table data-toolbar=${toolbar}>` // payload: #t onmouseover=alert(1) // result: <table data-toolbar=#t onmouseover=alert(1)> // Space ends the attribute value — onmouseover becomes a new attribute.

Why escapeHtml() Alone Fails for Case C

  • 1
    escapeHtml() only encodes: & < > " '
  • 2
    javascript:alert(1) contains none of those characters.
  • 3
    escapeHtml("javascript:alert(1)") returns "javascript:alert(1)" — identical output.
  • 4
    Browser sees valid href="javascript:alert(1)" and executes it as JS on click.
  • 5
    Fix: URL scheme allowlist first — reject anything that isn't http: or https:.
