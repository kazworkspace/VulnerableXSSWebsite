JS Context XSS

User input lands inside a <script> block. This is a different injection context — escapeHtml() is the wrong fix here. The correct tool is JSON.stringify() with extra encoding.

⚠
Root cause: User input is interpolated directly into a JavaScript string literal inside a <script> block. A " in the input closes the string — everything after it executes as code.

Why escapeHtml() Is the Wrong Fix Here

🔴 Context mismatch — wrong tool for the job
// Suppose we "fix" it with escapeHtml(): const color = escapeHtml(req.query.color); // Input: lightblue"; alert(1);// // Output: lightblue"; alert(1);// // Rendered script block: var bgColor = "lightblue&quot;; alert(1);//"; // In a <script> block, HTML entities are NOT decoded. // JavaScript sees: &quot; as literal characters, not ". // So the " injection is prevented... BUT: // ❌ Still fails for single-quote templates: // var x = '${color}'; + payload: '; alert(1);// // escapeHtml encodes ", not ' — single quote breaks out. // ❌ Still fails for non-string JS contexts: // var x = ${color}; (no quotes) — any JS expression works // ❌ Still fails for template literals: // var x = `${color}`; + payload: ${alert(1)} // escapeHtml doesn't encode $, {, } — template expression executes. // ✅ Use JSON.stringify() — it's designed for this context.

Code — Vulnerable vs Secure

🔴 VULNERABLE — /api/theme
const color = req.query.color || ''; // ❌ Raw interpolation in <script> block res.send(` <script> var bgColor = "${color}"; // payload: "; alert(1);// // result: var bgColor = ""; alert(1);//"; </script> `);
🟢 SECURE — /api/theme/safe
const color = req.query.color || ''; // ✅ JSON.stringify + encode < > / const safe = safeJsonForScript(color); res.send(` <script> var bgColor = ${safe}; // safe includes quotes + escaping </script> `);

Two Attacks in One Scenario

  • A
    String breakout: Payload "; alert(1);// — the " closes the string literal. alert(1) runs as top-level JS. // comments out the trailing ";.
  • B
    Script tag close: Payload </script><script>alert(1)// — the HTML parser closes the <script> block early when it sees </script>. A new script tag opens immediately after.
  • C
    safeJsonForScript() handles both: JSON.stringify escapes " as \", and extra encoding converts < → < so the HTML parser never sees a real </script>.
